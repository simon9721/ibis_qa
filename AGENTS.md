# AGENTS.md — IBIS QA Tool

Guidance for AI agents working in this repository.

## What this project is

A Python tool (`ibis_qa_tool/`) that evaluates IBIS `.ibs` model files
against the **IBIS Quality Specification v3.0** (68 checks across IQ
levels 1–4). It parses an `.ibs` file once into an `IBISFile` object, runs
all registered check modules against it, and produces text/JSON/Markdown/
HTML/spreadsheet reports — plus a reviewer workflow for the semi-auto and
manual checks that need human judgement.

The canonical source of truth for *what each check requires* is
`data/ibis_quality_spec_3_0.json`. Human-readable derivatives of that JSON
live in `docs/` and are regenerated by scripts in `tools/`.

## Core architecture rules (do not violate)

1. **Single parse**: `.ibs` files are tokenized once into an `IBISFile`.
   Checks must operate on that object — never re-read or re-parse the file.
2. **Plug-in checks**: each check or small family lives in its own file
   under `checks/`, subclassing `CheckModule` with `check_ids`, `iq_level`,
   `auto_class` set. `runner.py` discovers it automatically — **no
   registration step**. See `ibis_qa_tool/README.md` "Adding a New Check"
   for the template.
3. **Tolerances live in `config.py`** — never hard-code thresholds inside
   a check module.
4. **NA vs FAIL**: `NA` means the rule doesn't apply (e.g. bare-die
   components, input-only models, technology exceptions). It is not a
   failure and must not block an IQ level claim. See the "NA Handling"
   section of `ibis_qa_tool/README.md` before changing check logic that
   touches scope/applicability.
5. **`CheckResult.automation_class`** is `auto`, `semi_auto`, `manual`, or
   `optional`, and `review_required` flags items needing reviewer sign-off.
   Semi-auto checks should produce evidence (`WARN`/`review_required=True`),
   not hard `FAIL`, unless the spec clearly mandates a hard failure.
6. **Review overlays** (`*.review.json`) are a separate, additive layer —
   never mutate generated QA evidence to "fix" a review outcome; apply
   decisions via `review.py`.

## Working with the spec data

- `data/ibis_quality_spec_3_0.json` is the canonical source. If you change
  it, regenerate derivatives:
  ```powershell
  python tools/build_spec_data.py
  python tools/render_spec_tree.py
  python tools/render_automation_catalog.py
  python tools/render_level_catalog.py
  python tools/render_qa_methods.py
  ```
- For implementation details of any specific check (thresholds, PASS/FAIL/
  WARN/NA logic, known judgement calls), read
  `docs/implemented-checks.md` (auto/semi-auto) or `docs/manual-checks.md`
  (manual, out of scope for automation) **before** modifying a check.
- `docs/todo-review.md` and `changelog.md` track what's already been
  implemented/calibrated — check these before assuming something is unbuilt.

## Running and testing

No third-party dependencies (standard library only); Python 3.9+.
There is no automated test suite — verification is done by running the CLI
against the bundled sample files and checking output/compilation.

```powershell
# Sanity check: does everything still import/compile?
python -m py_compile ibis_qa_tool\*.py ibis_qa_tool\checks\*.py ibis_qa_tool\parser\*.py

# Run against the reference Micron files (from ibis_qa_tool/)
cd ibis_qa_tool
python ibis_qa.py ..\Micron\z41c-ibis\z41c.ibs --markdown --plot-dir ..\reports\micron\z41c_assets > ..\reports\micron\z41c.md
python ibis_qa.py ..\Micron\z41c-ibis\z41c.ibs --json > ..\reports\micron\z41c.json
python ibis_qa.py ..\Micron\z41c-ibis\z41c.ibs --spreadsheet ..\reports\micron\z41c.xlsx
```

After any change to a check module, parser, reporter, plotting, or
spreadsheet code, regenerate the Micron reports (`y32a.ibs`, `z41c.ibs`,
`z41c_it.ibs` under `reports/micron/`) and confirm the PASS/FAIL/WARN/NA/
ERROR counts move in the expected direction — `changelog.md` keeps a running
table of these counts per change for comparison. IBISCHK (check 2.1) is
auto-discovered at `ibischk721_win_64/ibischk7_64.exe`.

## Conventions

- This repo's instructions (CLAUDE.md global rules) say: write minimal
  comments, no speculative abstractions, no backwards-compat shims for
  unused code.
- Update `changelog.md` (dated entries, Added/Changed/Tested sections) and
  `ibis_qa_tool/README.md`'s check-coverage tables when adding/changing a
  check's behavior or status.
- Windows is the primary dev environment — use PowerShell syntax in
  examples/docs (the README already does this).
